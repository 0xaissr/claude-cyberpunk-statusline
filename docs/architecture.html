主題系統架構方案

選一個你覺得最合理的架構方向

A

Single Script + JSON Config

一個 statusline.sh 腳本 + 每個主題一個 JSON 檔。腳本讀取 config.json 裡指定的主題，從 JSON 取得顏色和符號來渲染。

cyberpunk-statusline/
  statusline.sh ← 唯一腳本，讀 config 渲染
  config.json ← wizard 產出（選了哪個主題、開關哪些區塊）
  themes/
    terminal-glitch.json ← 配色 + 符號 + 佈局定義
    neon-classic.json
    synthwave-sunset.json
    blade-runner.json
✓ 社群貢獻只要提交一個 JSON
✓ 邏輯集中，維護簡單
✗ 所有主題共用同一個渲染邏輯，彈性有限
B

Theme = Script + JSON

每個主題是一個資料夾，包含自己的 shell script 和 JSON 設定。主題可以完全自訂渲染邏輯。

cyberpunk-statusline/
  config.json ← wizard 產出
  core.sh ← 共用函式（make_bar, jq 解析等）
  themes/
    terminal-glitch/
      theme.json ← 配色 + 符號
      render.sh ← 自訂渲染（可 override）
    neon-classic/
      theme.json
      render.sh
✓ 主題可以有完全不同的渲染風格
✓ 進階使用者可寫自己的 render.sh
✗ 社群貢獻門檻較高（要寫 shell）
✗ 維護多個腳本，相容性風險高
C

Hybrid — JSON 優先 + 可選 Script Override

預設用 JSON 驅動（方案 A），但主題可以「選擇性」提供 render.sh 來覆蓋預設邏輯。兼顧簡單和彈性。

cyberpunk-statusline/
  statusline.sh ← 預設渲染引擎
  config.json ← wizard 產出
  themes/
    terminal-glitch.json ← 純 JSON，用預設引擎
    neon-classic.json
    custom-crazy/
      theme.json ← 配色定義
      render.sh ← 有這個檔就用它，沒有就用預設
✓ 簡單主題只要一個 JSON（低門檻）
✓ 進階主題可完全自訂渲染
✓ 向後相容，漸進式複雜度
✗ 稍微複雜一點的 statusline.sh（要判斷 render.sh 是否存在）
